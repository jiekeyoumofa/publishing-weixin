# Skill: oral-to-article

# oral-to-article

把口述 / 草稿 / 散乱 bullet / 语音转写，转成一篇可以独立阅读、结构完整的文章（或几个打磨过的片段）。

只做一件事：轻润色，不重写。保留作者的语气和节奏。

## Core Principle

用户的思路和表达方式是文章的灵魂。润色只做四件事：

- 修明显错字和重复字
- 调整段落（每段 1–3 句）
- 抚平特别拗口的句子（保守，能不动就不动）
- 准备配套标题候选 + 摘要

改动尺度：改动的字数不超过原文的 5%，超过就退回去。

## Inputs to collect

- 内容本身（草稿 / bullet / 语音转写 / 一大段未分段的文字）
- 分支判断：用户到底想要"一篇文章"还是"几个独立想法"（见 Step 0.5）。只问这一次。

## Procedure

### 长度与例子的硬约束

- 默认 800–1000 字。2000 字算超长，必须有特别理由（系列长文、技术深度文）才能放过。
- 第一稿就按这个预算写，不要写完再砍——按预算写出来的文章节奏紧，砍出来的文章会残留拼接感。
- 写到例子段时过一遍这把尺：
  - 这个例子是真的具体（真事 / 真人 / 真数字 / 真细节），还是为了演示框架编出来的？
  - 后者直接删。
- 优先保留：开头反差 / 钩子 + 核心框架 + 1 句点睛 + 软着陆结尾。
- 优先砍掉：演示性例子、重复阐释、"怎么用 / 入口在哪"这类 instructional 段落。

### Step 0：接收输入

用户可能给以下形式：

- 完整草稿（最常见）
- 几段散乱的思路 / bullet points
- 一段长文字，没有分段
- 语音转写（可能有错字、重复）

如果输入太散，只问一个问题："这是想写一篇文章，还是几个独立想法？"——只问这一次。

### Step 0.5：分流决策

只问一次，不追问第二遍。

- **分支 A：一篇文章**（默认路径）→ 进入 Step 1 轻润色，按 800-1000 字、标题 ×3、摘要 50-80 字的标准流程走。
- **分支 B：几个独立想法** → 不强行合并，按"片段"输出。

分支 B 的处理细节：

- 字数：200-600 字（软上限，可短不可长）
- 标题候选：1 个就够（不是 3 个）
- 摘要：30-50 字
- 可省字数闸门（短到不需要跑）
- 文件组织：
  ```
  articles/YYYY-MM-DD-{slug}/
  ├── ideas/
  │   ├── idea-1-{mini-slug}/
  │   │   ├── fragment.md      # 润色后的片段
  │   │   ├── title.md         # 1 个标题候选
  │   │   └── summary.md       # 30-50 字摘要
  │   ├── idea-2-{mini-slug}/
  │   └── idea-N-{mini-slug}/
  ```
- 片段继承 Step 1 所有核心规则（保留语气、改动 ≤5%、不补例子、不改顺序），但去掉：
  - 不强制段落 1-3 句
  - 不跑字数闸门
  - 不强制"例子段是真事"检查（没篇幅写例子）
- 结果展示用下面这个统一格式（汇报本身就是触发后续决策的钩子）：

  ```
  我处理了 N 个独立想法：

  📄 idea-1：{mini-slug}
    - 片段（{字数}字）：{首句前 30 字}…
    - 标题候选：{标题}
    - 摘要：{30-50 字摘要}
    - 改动：{N 处，主要改动 1-2 行}  ← 仅 >3 处时列

  📄 idea-2：{mini-slug}
    - 片段（{字数}字）：{首句前 30 字}…
    - 标题候选：{标题}
    - 摘要：{30-50 字摘要}
    - 改动：{N 处，主要改动 1-2 行}  ← 仅 >3 处时列

  📄 idea-3：{mini-slug}
    - 片段（{字数}字）：{首句前 30 字}…
    - 标题候选：{标题}
    - 摘要：{30-50 字摘要}
    - 改动：{N 处，主要改动 1-2 行}  ← 仅 >3 处时列

  文件落在：articles/{date}-{slug}/ideas/

  接下来怎么办？
    A) 合并：找共同主题，串成 1 篇完整文章
    B) 独立：每片段各用（tweet / 多篇短文 / 素材库）
  ```

- "首句前 30 字…" 用作预览，不贴全文（用户要看全文自己打开 fragment.md）。
- 每个片段改动 ≤ 3 处省略 diff 行，> 3 处才列。
- "接下来怎么办"必须给两个出口，不催第二遍。

不强行合并 / 独立的原因：

| 强行做的选项 | 问题 |
| --- | --- |
| 强行合成一篇 | 几个无关想法硬塞一起 → 结构松散，"AI 味"重，违反「不重写」原则 |
| 各自写完整 800-1000 字 | 用户可能只是想要 tweet / bullet 库，杀鸡用牛刀 |
| 再追问一次"要怎么合" | 违反"只问一次"约束，反而显得啰嗦 |

### Step 1：轻润色（核心转换）

打开一个 markdown 文件，把用户的内容粘进去。只做下面这些：

#### 1.1 修错字

- "的得地" 误用 → 根据语法判断
- 重复字 "我我" / "是是" / "了了" → 删一个
- 同音字 "在 vs 再"、"做 vs 作" → 看上下文
- 其他明显的同音错字 → 按上下文订正

#### 1.2 段落切分

分段时机：

- 一句话讲完一个意思，下一句换主语 → 分段
- 出现"但是"、"不过"、"所以"、"后来"在句首 → 考虑分段
- 一段超过 80 字 → 找最近的句号分
- 目标节奏：每段 1–3 句

不要分段：

- 排比句、列举（保持节奏）
- 对话（按对话格式）

#### 1.3 拗口处最小改动

- 改动后语气变了，宁可不改
- 不要把口语改成书面语
- 不要删口语句尾"吧 / 呢 / 啊"

#### 1.4 标点统一

- 中文用全角逗号句号
- 不动作者的标点偏好（哪怕略奇怪）

#### 1.5 保留原开头和结尾

- 这是作者的标志性特征
- 不要"提升"开头
- 不要在结尾加"欢迎关注"、"点赞在看"这类作者没写的东西

### Step 2：标题候选

给 3 个标题候选：

- A) 直白型：直接说文章讲什么
- B) 故事型：从一个场景或冲突切入
- C) 原文金句：从草稿里摘最有味道的一句

不做：标题党、夸张、"震惊"、"必看"。

### Step 3：摘要（50–80 字）

- 不是文章第一段的复制
- 一句话说清楚读者会获得什么
- 用作者的语气，不是营销腔

### Showing the Diff

每次润色完，先告诉用户改了什么，再问要不要继续：

```
我改了 7 处：
1. L3: "我我觉得" → "我觉得"（重复字）
2. L8: 长段落 (120字) 拆成两段
3. L15: "通过…的方式" → "用…"（口语化保留）
...
要看完整结果吗？
```

如果改动 ≤ 3 处，可以直接给完整结果，不用列 diff。

## Output contract

### 分支 A：一篇文章

- 一篇 800-1000 字的结构完整文章（超不超过 1200）
- 3 个标题候选
- 50-80 字摘要
- 改动 ≤ 原文 5% 的 diff（≤ 3 处可直接给全文）
- 输出形式：聊天内贴 markdown，或写入 `articles/YYYY-MM-DD-{slug}/article.md`

### 分支 B：几个独立想法

- 每个想法独立成片段，文件落在 `articles/YYYY-MM-DD-{slug}/ideas/idea-N-{mini-slug}/`
- 每个片段 ≤ 600 字
- 每个片段 1 个标题 + 1 个 30-50 字摘要
- 汇报时给两个后续出口（合并 / 独立），不替用户做决定

## Anti-Patterns

| 不要 | 原因 |
| --- | --- |
| 把"我觉得"改成"笔者认为" | 改变了作者身份 |
| 加"小标题"打断行文 | 读者不需要导航 |
| 把口语句尾"吧/呢/啊"删掉 | 删掉就不是这个人写的了 |
| 在结尾加"欢迎关注"、"点赞在看" | 作者会自己决定要不要 |
| 把"今天"改成具体日期 | 作者用"今天"是有意为之 |
| 自己加举例 / 引用 / 数据 | 这是写作，不是补全 |
| 改动后没给 diff，直接全文输出 | 用户看不见你改了什么 |
| 文章超过 1500 字 | 多半是某段空例子在撑场。砍掉那段，看会不会自然回到 1000 字 |
| 为演示框架编一段完整例子 | 正文只留点睛，不复述结构 |
| 加 AI 味儿的连接词 | "首先"、"其次"、"综上所述"、"总而言之"、"值得注意的是" |
| 把口语改成书面语 | 改了就换了个人 |
| 重新组织段落顺序 | 顺序是作者定的 |

## Failure handling

- **输入太散无法判断意图** → 问一次"一篇还是几个独立想法"，答不上来就默认走分支 A（用户最常见的诉求）。
- **改动会超过原文 5%** → 退回去，告诉用户"要保留你的语气/顺序必须重写，要不要这样？"。不要硬上。
- **文章超 1000 字但没到 1200** → 警告一次（"超 1000 字了，看看要不要砍"），不强制砍；超 1200 才强制按"硬约束"段砍例子。
- **例子段都是编的** → 全部删除，按"硬约束"段的尺子过一遍。
- **作者的语气和"标准文章写法"冲突** → 以作者为准。这是个"轻润色"skill，不是"提升文笔"skill。
- **用户对 diff 不满意** → 回滚到上一版，问具体哪条改得不对，不要瞎改第二条。
- **分支 B 跑完用户没选 A/B** → 不催第二遍，把文件路径留给用户，等他回头处理。

## Examples

### Example 1：分支 A——完整草稿润色

**Input**（用户）：
> 我我做产品做了 8 年了，最近有个很深的感受。就是大多数公司都在追求那种所谓的"颠覆式创新"，但其实呢，你去看那些真正活下来的公司，他们做的东西都很土。土不是贬义，是说你看那些东西好像没有那种特别 fancy 的感觉。但是它解决了问题，所以它活下来了。

**Output**（片段）：

> 我做产品做了 8 年了，最近有个很深的感受。
>
> 大多数公司都在追求那种所谓的"颠覆式创新"，但你去看那些真正活下来的公司，他们做的东西都很"土"。
>
> 土不是贬义，是说那些东西好像没有特别 fancy 的感觉。但是它解决了问题，所以它活下来了。

**Diff**：
```
我改了 2 处：
1. L1: "我我" → "我"（重复字）
2. L2: 长段拆成 3 段（每段一个意思）
```

**标题候选**：
- A) 直白型：活下来的产品，都有点"土"
- B) 故事型：我做了 8 年产品，才看懂"土"的价值
- C) 金句型：它解决了问题，所以它活下来了

**摘要**：做了 8 年产品，越来越相信"土"的东西反而活得久。颠覆式创新是个伪命题，能解决问题的就是好产品。

### Example 2：分支 B——几个独立想法

**Input**（用户）：
> 我有 3 个想法：1）现在大家都在讲 AI agent，但大多数 demo 都很浅；2）周会和文档之间的关系可以重新想一下；3）离职率高的公司，HR 流程一定有问题

**Output**：

```
我处理了 3 个独立想法：

📄 idea-1：ai-agent-demo-浅
  - 片段（280 字）：现在大家都在讲 AI agent，但大多数 demo...
  - 标题候选：现在的 AI agent demo，为什么都这么浅？
  - 摘要：3 分钟讲清 agent demo 浅在哪，缺的不是技术，是真实场景

📄 idea-2：周会与文档
  - 片段（420 字）：周会和文档之间的关系可以重新想一下...
  - 标题候选：周会太多，是因为文档没写对
  - 摘要：周会和文档不是替代关系，是同一个事情的两种表达

📄 idea-3：离职率与 HR
  - 片段（360 字）：离职率高的公司，HR 流程一定有问题...
  - 标题候选：HR 流程跑顺了，离职率自然降
  - 摘要：离职率是结果，HR 流程才是原因，反过来推一下就懂了

文件落在：articles/2026-06-20-three-ideas/ideas/

接下来怎么办？
  A) 合并：找共同主题，串成 1 篇完整文章
  B) 独立：每片段各用（tweet / 多篇短文 / 素材库）
```

## Done When

### 分支 A：一篇文章

- [ ] 改动 ≤ 原文 5%
- [ ] 字数 800–1000（超不超过 1200）
- [ ] 每段 1–3 句，无超 80 字长段
- [ ] 标题给了 3 个候选
- [ ] 摘要 50–80 字
- [ ] 已展示 diff（或 ≤ 3 处改动直接给全文）
- [ ] 用户没说"再改改"

### 分支 B：几个独立想法

- [ ] 每个想法独立成片段，文件落在 `articles/YYYY-MM-DD-{slug}/ideas/`
- [ ] 每个片段 ≤ 600 字
- [ ] 每个片段给了 1 个标题 + 1 个 30-50 字摘要
- [ ] 已给用户两个后续出口（合并 / 独立），未替用户做决定
- [ ] 用户没说"再改改"
